# 反思题

> V3 运单异常工单审批系统 — 考试反思题

---

## 1. 如果题目直接给出了精确的审批金额阈值和超时时长，你的架构设计会有什么不同？

**回答**：

如果题目直接给出了精确数值，架构设计的**核心不会变**——仍然会使用可配置的规则引擎（`approval_level_rules` 表 + `timeout_rules` 表），因为可配置性是"规则引擎"的核心要求，不应因为数值已知就退化为硬编码。

但会有以下不同：

1. **初始化逻辑简化**：不需要花时间推敲阈值合理性，直接按题目给的值写入默认规则
2. **假设说明文档简化**：不需要解释"为什么是 ¥500"或"为什么是 60 分钟"
3. **测试用例明确**：可以直接用题目给的阈值设计边界测试（¥499 走一级，¥500 走二级）
4. **少了业务思考环节**：自行设定阈值的过程实际上在倒逼开发者思考业务合理性，这是有价值的

**核心不变的是**：无论数值是谁给的，都应通过配置表管理而非硬编码，这是 V2 延续下来的设计理念。

## 2. 状态机设计的优缺点

**问题**：当前工单状态机采用 `pending → level2 → approved → closed` 的四状态设计，请分析这种设计的优缺点。

**回答**：

- **优点**：状态数量少，流转清晰；金额阈值驱动二级审批，规则简单；拒绝后回到 pending 支持重提。
- **缺点**：
  - 状态抽象层次不够。比如 `pending` 和 `level2` 实际都是"审批中"的子状态，但平铺在一起。
  - 缺少"撤回"状态。上报人提交后无法主动撤回工单。
  - 缺少"挂起/暂缓"状态。实际业务中有些工单需要等待证据补充。
  - 状态机未持久化到数据库，服务重启后状态丢失。
- **改进建议**：引入"活动状态"和"终态"的概念分层；添加 `suspended`（挂起）、`withdrawn`（撤回）、`archived`（归档）状态。

## 3. 你认为本题里"审批状态变更"与"库存/赔付联动"之间的一致性，最大的风险点在哪里？如果不用数据库事务，你会怎么处理？

**回答**：

**最大风险点**：审批通过 → 执行联动之间的"中间态"——审批状态已变更为"执行中"，但库存扣减/赔付生成因为网络抖动、服务崩溃等原因没有完成。

具体场景：
1. 审批人点击"通过"→ 工单状态改为"executing" → 写审批记录成功 → 库存扣减失败
2. 此时工单显示"执行中"，但实际什么都没执行，也没人能感知

**如果不用数据库事务**，我会采用**补偿事务（Saga 模式）**：

1. **状态机驱动 + 异步任务**：
   - 审批通过后立即写入一条"待执行任务"记录（`execution_tasks` 表）
   - 后台任务逐条消费任务：扣库存 → 生成赔付 → 标记完成
   - 每个子任务独立执行，失败则重试

2. **补偿机制**：
   - 如果某个子任务在重试上限后仍然失败，执行补偿操作（回滚已完成的前置步骤）
   - 例如：库存已扣但赔付生成失败 → 补偿回滚库存（库存加回）
   - 补偿失败则记录到"待人工处理"队列

3. **幂等性保证**：
   - 每个子任务有唯一 `task_id`
   - 执行前检查是否已完成（`status = 'done'`），防止重复执行

**对比数据库事务**：事务是最简单的方案（ACID 保证），但跨表事务在分布式场景下有限制。Saga 模式更复杂但更具扩展性。

## 4. 并发控制的充分性

**问题**：当前系统通过内存状态机的同步性来实现并发控制。你认为这种方案在什么场景下会失效？如何改进？

**回答**：

- **失效场景**：
  - 多进程/多实例部署时，内存状态不共享，两个不同进程可以同时审批同一个工单。
  - Node.js 虽然是单线程事件循环，但 `await` 会让出控制权，如果两个请求在同一个 tick 内读取到相同状态，仍可能产生竞态条件。
- **改进方案**：
  - 短期：引入 Node.js `async-mutex` 锁，按 ticket_id 做互斥。
  - 长期：工单持久化到 PostgreSQL，使用 `SELECT ... FOR UPDATE` 行锁，配合乐观锁版本号（`version` 字段）。
  - 分布式：引入 Redis 分布式锁（Redlock）或使用数据库事务隔离级别 SERIALIZABLE。

## 5. 面对一句话级别的模糊需求，你倾向于"先问清楚再动手"还是"先做出一个版本再迭代"？为什么？

**回答**：

我倾向于**"先做出一个最小可行版本，再基于反馈迭代"**，理由如下：

1. **模糊需求往往来自需求方自己也说不清楚**：很多时候产品经理的"模糊"不是因为不想说清楚，而是因为业务场景本身还在探索中。一个可运行的 MVP 比一份完美的需求文档更能帮助双方对齐认知。

2. **代码即文档**：一个有实际 UI 的原型比任何文字描述都直观，需求方看到界面后往往会说"哦不对，我想的其实是 XX"，这种反馈比事前追问更高效。

3. **时间成本权衡**：等需求方回复可能需要几天（实际考试场景中根本没机会问），先做出一个版本至少可以展示技术能力，后续再调整。

**但有限定条件**：
- 核心数据模型（表结构）和状态机（状态转换规则）必须在一开始就深思熟虑，因为这些是"地基"，后期改动成本极高
- UI 交互细节、阈值数值等可以先用合理默认值，后续调整
- 必须在假设说明文档中明确记录所有假设，避免后续扯皮

## 6. 赔偿方向的合理性

**问题**：当前系统在 `tickets/store.ts` 和 `v3-store.ts` 中对同一异常类型的赔付方向定义不一致（如 `damaged` 在 v3-store 中为"核销"而在 tickets/store 中为"赔付客户"）。你认为哪个更合理？为什么？

**回答**：

- **赔付客户更合理**（`tickets/store.ts` 的选择）：破损通常是物流过程中的责任，应由物流方赔付客户，再内部追责。从客户体验角度出发，破损件应当"先赔付客户"。
- **核销也有道理**（`v3-store.ts` 的选择）：如果破损商品经过品控确认可以打折销售或内部消化，走核销流程更符合成本控制逻辑。
- **正确做法**：应当根据破损责任归属（物流责任 / 商家责任 / 客户责任）动态确定赔付方向，而非一刀切。责任归属字段可以作为一个扩展点。

## 7. 如果产品经理后续要求新增第六种异常类型（比如"客户投诉服务态度"，不涉及货品本身、不涉及库存联动），你现在的表结构和状态机需要改动哪些地方？改动成本大不大？

**回答**：

| 改动点 | 改动内容 | 成本 |
|--------|----------|------|
| `exception_tickets` 表 | 无需改动 | 零 |
| `EXCEPTION_TYPES` 常量 | 新增 `"service_complaint"` | 极小 |
| 前端类型选择 UI | 新增选项 | 极小 |
| 执行联动逻辑 | 新增 case：无需赔付/无需库存联动 | 小 |
| 赔付记录表 | 无需改动（不涉及赔付则不创建记录） | 零 |
| 库存日志表 | 无需改动 | 零 |
| 状态机 | 无需改动（复用现有审批流程） | 零 |

**改动成本评估**：**非常小**。

原因在于：
1. 异常类型是**开放枚举**，不是硬编码的固定值，新增类型只需加一个 case
2. 状态机与异常类型解耦——审批流程对所有类型通用
3. 执行联动是按类型分发的 switch-case，新增一个"什么都不做"的 case 即可
4. 表结构不需要变更，因为赔付和库存是可选的（不是所有异常都需要）

**这证明了当前架构的扩展性良好**。

## 8. 如果工单量从 200 条增长到 20 万条，你的系统里最先撑不住的环节是哪个？你会怎么处理？

**回答**：

**最先撑不住的环节**：**工单列表的分页查询**（全表扫描 + JOIN）。

分析：
1. **`SELECT * FROM exception_tickets ORDER BY created_at DESC LIMIT 20 OFFSET X`**：当 offset 很大时（比如第 10000 页），PostgreSQL 需要扫描并跳过前 199980 行，性能急剧下降
2. **列表页的 COUNT(*)**：全表计数在 20 万行时可能耗时数秒
3. **定时任务的全表扫描**：检查超时工单如果做 `SELECT * WHERE status IN ('pending','level1','level2') AND due_at < NOW()`，在有索引的情况还好，但 20 万行仍可能有压力

**处理方案**：

1. **游标分页替代 offset 分页**：
   ```sql
   SELECT * FROM exception_tickets
   WHERE created_at < $lastCursor
   ORDER BY created_at DESC LIMIT 20
   ```
   使用 `created_at` 作为游标，避免大 offset

2. **COUNT 使用估算值**：
   - 使用 PostgreSQL 的 `pg_class.reltuples` 获取近似行数
   - 或者缓存 COUNT 结果，每 5 分钟刷新一次

3. **超时检查优化**：
   - 在 `(status, due_at)` 上建复合索引
   - 改为事件驱动：工单创建时计算超时时间，用延迟队列触发（如 Redis 的 `ZSET` 或 BullMQ）
   - 减少定时全表扫描的频率

4. **读写分离**：
   - 列表查询走只读副本
   - 审批操作走主库

5. **归档历史数据**：
   - 已完成/已关闭超过 90 天的工单迁移到归档表
   - 列表默认只查询近 90 天数据

## 9. 快照降级策略的边界条件

**问题**：V3 的运单数据获取采用"V2 实时拉取 → 本地快照 → 硬编码 fallback"三级降级。请分析各层降级的边界条件和失效风险。

**回答**：

- **第一层 V2 实时拉取**：
  - 失效条件：V2 服务宕机、网络分区、超时（8秒）、鉴权失败
  - 风险：V2 部分可用时（连接成功但查询超时）无法降级（当前实现中只有在 `catch` 块才降级）

- **第二层 本地快照**：
  - 失效条件：快照表不存在、首次同步未完成就 V2 故障、快照数据过期
  - 风险：**数据时效性问题** — 快照是上次同步时的数据，V2 新增的运单无法反映。建议增加 `synced_at` 过期策略。

- **第三层 硬编码 fallback**：
  - 失效条件：代码中的测试数据不匹配实际业务
  - 风险：**数据准确性为零** — 硬编码运单与实际运单完全无关，可能导致工单创建失败或数据混乱。

- **改进建议**：
  - 快照增加 TTL（如 24 小时过期）
  - V2 连接成功但返回错误的中间态也要降级（不只是 catch）
  - 硬编码 fallback 应该至少返回空数组而非假数据，让上层感知到数据不可用

## 10. 你在给 V2 新增对外接口时，如何保证不破坏 V2 现有的调用方？如果 V2 的运单金额字段类型从 `int` 改成 `decimal`，你会如何处理 V3 侧的兼容？

**回答**：

### 保证不破坏 V2 现有调用方

1. **接口路径隔离**：新增的对外接口使用独立路径前缀 `/api/waybills/`，与 V2 原有内部接口（如 `/api/rules/`、`/api/parse/`）完全隔离，不会冲突

2. **字段向后兼容**：
   - 新增字段使用可选类型（`?`），不要求调用方一定传入
   - 不删除现有字段，不改变现有字段的类型和语义
   - 响应 JSON 只增加字段，不减少字段

3. **版本化考虑**：
   - 当前 V1 接口不设版本号（路径为 `/api/waybills/sync`）
   - 如果后续需要 breaking change，使用 `/api/v2/waybills/sync` 路径
   - V1 接口保留至少 2 个版本周期

4. **接口测试**：
   - 新增接口在部署前做回归测试，确保 V2 原有功能不受影响
   - 接口契约通过文档明确约定

### V2 金额字段 `int` → `decimal` 的兼容处理

**V3 侧改动**：

1. **快照表已兼容**：V3 的 `waybill_snapshots` 表使用 `NUMERIC` 类型存储金额，天然兼容 `int` 和 `decimal`

2. **前端展示已兼容**：V3 前端使用 `parseFloat()` 和 `.toFixed(2)` 展示金额，不依赖整型假设

3. **审批规则已兼容**：`approval_level_rules` 使用 `NUMERIC` 类型的 `min_amount`/`max_amount`，与 V2 类型变更无关

4. **接口同步日志监控**：如果 V2 返回的 `amount` 字段格式变化，V3 的接口日志会记录响应内容，便于排查

**总结**：由于 V3 从一开始就设计为通过接口与 V2 交互（而非直连数据库），且使用了灵活的数据类型（NUMERIC、JSONB），V2 的类型变更对 V3 的影响极小。

## 11. 权限模型的安全隐患

**问题**：当前系统通过 `operator.includes("supervisor")` 判断放行权限。请分析这种权限模型的安全隐患。

**回答**：

- **安全隐患**：
  - **字符串包含匹配过于宽松**：任何包含 "supervisor" 子串的名字都能通过，如 `"not_a_supervisor_but_hacker"` 也能放行。
  - **无签名验证**：前端传入的 `operator` 字符串可以被任意伪造。
  - **无角色注册机制**：没有用户-角色的映射表，无法管理权限变更。
- **改进方案**：
  - 使用基于角色的访问控制（RBAC），用户-角色-权限三层模型
  - JWT Token 中携带用户角色，服务端解析 Token 获取角色
  - 配置 `allowedRoles: string[]` 用精确匹配替代字符串包含

**回答**：

- **未覆盖场景**：

  - 赔付金额计算与工单金额不一致的场景
- **建议补充**：添加混沌工程测试（随机注入网络故障、延迟），验证降级策略的鲁棒性。

## 14. 内存状态机 vs 数据库持久化

**问题**：当前 V3 将工单、审批记录、扫描记录存储在内存中。如果系统部署到生产环境，你认为必须将哪些数据持久化到数据库？为什么？

**回答**：

- **必须持久化的数据**：
  - **工单** — 核心业务实体，需要审计追踪
  - **审批记录** — 审批链需要不可篡改的留痕
  - **扫描记录** — 品控过程需要溯源
  - **赔付记录** — 涉及财务，必须有持久化保障
- **可以保留在内存中的数据**：
  - 计数器（如 `retry_count`）— 可以每次从数据库计算
  - 临时锁状态 — 可以用 Redis 处理
- **迁移策略**：先建表 → 双写（内存+数据库）→ 灰度切读 → 下线内存存储。
