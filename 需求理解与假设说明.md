# 需求理解与假设说明

> V3 运单异常工单审批系统 — 需求理解、假设与留白点说明

---

## 一、需求理解

### 1.1 系统定位

V3 系统是一个面向物流/仓储场景的 **运单异常工单管理与审批系统**，作为 V2（万能导入系统）的下游消费端，负责将 V2 导入的运单数据进行品控扫描、异常识别、多级审批和赔付联动处理。

### 1.2 核心业务场景

| 场景 | 说明 |
|------|------|
| **手动上报异常** | 仓管/品控人员发现运单异常（丢件、破损、短少、错件）后手动创建工单 |
| **扫描品控** | 扫描枪扫码核对 SKU 数量/规格/破损情况，不通过时自动创建工单并暂扣 |
| **分级审批** | 低金额工单一级审批直达，高金额（>500元）需二级审批 |
| **异常回写** | 审批结果回写 V2 系统，标记运单存在未闭环异常工单 |
| **赔付联动** | 审批通过后自动生成赔付/补货/换货方向记录 |

### 1.3 与 V2 的关系

V2 是"万能导入系统"，负责文件智能解析、运单数据入库。V3 是"异常管理审批系统"，通过 API 接口从 V2 拉取运单数据到本地快照，独立完成后续的工单管理流程。

## 二、九项关键留白规则的补充设定

### ① 分级审批金额阈值

| 审批层级 | 金额区间 | 说明 |
|----------|----------|------|
| 一级审批 | ¥0 - ¥499.99 | 小额异常仅需一级审批人确认 |
| 二级审批 | ¥500 及以上 | 涉及较大金额需二级审批人复核 |

**设定依据**：参考物流行业惯例，¥500 是一个常见的"小额快速处理"与"大额审批"的分界线。同时参照电商物流中常见的客单价区间（¥200-¥500），将阈值设为 ¥500 可以确保大多数常规异常在一级审批就能处理，而金额较大或涉及赔付的异常需要更高级别审批，降低风险。

**可配置性**：阈值存储在 `approval_level_rules` 表中，支持按金额区间和异常类型灵活配置，不硬编码。

---

### ② 审批超时时长

| 审批阶段 | 超时时长 | 超时动作 |
|----------|----------|----------|
| 待审批（pending） | 60 分钟 | 自动升级到二级审批 |
| 一级审批中（level1） | 120 分钟 | 自动升级到二级审批 |
| 二级审批中（level2） | 240 分钟 | 自动驳回，工单回到待审批状态 |

**设定依据**：
- 待审批 60 分钟：工单创建后应尽快分配审批人，1 小时内无人认领说明分配机制有问题
- 一级审批 120 分钟：给一级审批人足够的处理时间，但不应超过半个工作日
- 二级审批 240 分钟：二级审批人通常是更高级管理者，会议较多，给予半天时间
- 超时后升级而非直接关闭：保证异常最终得到处理

---

### ③ 重新提交次数上限

| 参数 | 数值 | 说明 |
|------|------|------|
| 最大重提次数 | 3 次 | 被拒绝后可重新提交，超过上限自动关闭 |

**设定依据**：
- 3 次是业务系统中的常见上限（如审批驳回、密码重试等）
- 超过 3 次说明工单信息可能存在根本性问题，继续循环无意义
- 关闭后由人工介入判断是否需要新建工单

---

### ④ 物流异常类型与下游动作映射

| 异常类型 | 执行动作 | 赔付方向 | 库存联动 | 说明 |
|----------|----------|----------|----------|------|
| 丢件（lost） | 理赔 | 赔付客户（to_customer） | 回滚原运单库存 | 确认丢件后赔款+释放预留库存 |
| 破损（damaged） | 理赔 | 赔付客户（to_customer） | 退货入库（如可回收） | 根据破损程度决定赔付金额 |
| 客户拒收（rejected） | 退货入库 | 赔付客户（to_customer） | 退货入库 +库存 | 货物退回仓库，如产生额外费用赔付 |
| 超时未签收（timeout） | 催收/理赔 | 视情况 | 不变 | 超时后可能重新派送或理赔 |
| 地址错误（wrong_address） | 重新发货 | 无（重新发货） | 扣减库存重新发货 | 修正地址后重新发出 |

**设定依据**：参考主流物流平台（顺丰、京东物流）的异常处理流程，每种异常类型的处理链路不同，且赔付方向必须区分。

---

### ⑤ 角色权限划分

| 角色 | 权限范围 | 说明 |
|------|----------|------|
| admin（管理员） | 全部权限 | 可操作所有功能 |
| qc_supervisor（品控主管） | 扫描品控 + 快速放行 + 查看工单 | 可对品控暂扣执行快速放行 |
| level1_approver（一级审批人） | 一级审批 + 查看工单 | 审批金额 < ¥500 的工单 |
| level2_approver（二级审批人） | 二级审批 + 查看工单 | 审批金额 >= ¥500 的工单 |
| reporter（上报人） | 异常上报 + 查看自己上报的工单 | 不能审批自己的工单 |
| operator（操作员） | 扫描录入 + 查看扫描记录 | 执行日常扫描操作 |

**角色互斥规则**：
- 上报人不能审批自己提交的工单（后端强制校验）
- 同一人可以拥有多个角色，但自批自核被禁止
- 品控主管与审批人不重叠（品控主管负责误判复核，审批人负责业务审批）

---

### ⑥ V2 数据同步频率与一致性策略

| 策略 | 说明 |
|------|------|
| 同步触发方式 | **混合模式**：异常上报时实时拉取最新运单信息；定时任务（每 30 分钟）批量同步增量数据 |
| 快照数据标注 | 工单详情页明确标注数据来源："实时获取自 V2" 或 "本地缓存，同步于 XX 时间" |
| V2 不可用降级 | 使用本地快照表最新数据，前端明确提示"数据可能非最新" |
| V2 数据变更感知 | 每次异常上报时做一次实时接口校验，发现不一致则更新快照并记录差异日志 |
| 一致性冲突处理 | 定期对账（记录差异），由人工介入判断；不自动覆盖 V2 数据 |

**设定依据**：
- 混合模式平衡了实时性和性能：关键动作（异常上报）实时校验，日常展示使用缓存
- 30 分钟同步间隔是"接近实时"与"不压垮 V2"的折中
- 明确标注数据来源符合审计要求

---

### ⑦ 品控暂扣超时时长

| 参数 | 数值 | 说明 |
|------|------|------|
| 品控暂扣超时 | 30 分钟 | 独立于审批超时 |

**为何独立于审批超时**：
品控暂扣意味着货物被锁定在仓库，产生仓储成本（压仓损耗）。审批环节的超时是管理效率问题，而品控暂扣超时是物理成本问题——货物压仓每分钟都在产生成本。

**数值确定依据**：
- 品控暂扣发生在仓库作业现场，操作人员和品控主管都在现场
- 30 分钟足够品控主管到达现场复核，但不会让货物长时间被锁
- 远短于审批超时（60/120/240 分钟），因为品控问题不需要走完整审批流程
- 超时后强制升级到二级审批（而非自动放行），防止问题货物流入下游

### ⑧ 品控规则触发阈值

| 规则 | 触发条件 | 严重度 | 审批级别 |
|------|----------|--------|----------|
| 数量差异 > 10% | 实际数量与期望数量偏差超过 10% | 中（medium） | 一级 |
| 数量差异 > 30% | 实际数量与期望数量偏差超过 30% | 高（high） | 二级 |
| 破损等级 >= 2 | 货物破损等级达到 2 级（轻微破损） | 中（medium） | 一级 |
| 破损等级 >= 4 | 货物破损等级达到 4 级（严重破损） | 严重（critical） | 二级 |
| 规格不符 | SKU 规格与运单记录不匹配 | 中（medium） | 一级 |
| 标签错误 | SKU 标签信息错误 | 高（high） | 二级 |
| 批次异常 | 货物批次号异常 | 严重（critical） | 二级 |

**设定依据**：
- 10% 数量差异：物流行业通常允许 ±5% 的合理误差，10% 作为告警阈值合理
- 30% 数量差异：差异超过三成通常意味着系统性错误，需高级审批
- 破损等级 1-5：等级 2（轻微磕碰/包装破损但货物完好）是可接受的，等级 4（货物实质性损坏）不可接受
- 所有阈值存储在 `qc_rules` 表中，可随时调整

---

### ⑨ 品控主管角色权限边界

| 权限 | 品控主管 | 一级审批人 | 二级审批人 |
|------|----------|-----------|-----------|
| 扫描操作 | ✓ | ✗ | ✗ |
| 品控快速放行 | ✓（专属） | ✗ | ✗ |
| 一级审批 | ✗ | ✓ | ✓ |
| 二级审批 | ✗ | ✗ | ✓ |
| 查看工单 | ✓ | ✓ | ✓ |
| 异常上报 | ✗ | ✗ | ✗ |

**品控主管权限边界**：
1. **唯一拥有快速放行权限**：品控主管可以在认定扫描结果为误判时，直接解锁批次并关闭工单，绕过审批流程
2. **不能审批业务工单**：品控主管的职责是品控复核，不是业务审批，两者职责分离
3. **快速放行必须留痕**：填写复核原因，记录操作人和时间，不允许静默放行
4. **一人可以兼任多角色**：如果同一人同时是品控主管和审批人，系统不禁止，但自批自核（审批自己上报的工单）仍然被禁止

## 三、关键假设与留白点

### 3.1 数据同步与快照

**假设**：V3 通过主动拉取（Pull）方式从 V2 获取运单数据，首次同步成功后自动缓存到本地 `waybill_snapshots` 表。

**降级策略**：
1. **优先 V2 实时拉取**（8秒超时）
2. **快照兜底**（V2 不可用时读取本地快照）
3. **硬编码 fallback**（快照也无数据时的最后保障）

**留白说明**：当前为单向同步（V2→V3），V3 的审批结果通过 `/api/waybills/exception-status` 回写到 V2。真实场景可能需要双向同步或消息队列保证最终一致性。

### 3.2 并发控制

**假设**：审批操作需要互斥保护，同一工单不能同时被两人审批。

**实现**：使用幂等性检查（同一审批人重复操作返回 409）+ 状态机互斥（已完结工单不可再操作）。

**留白说明**：当前使用内存状态机，单进程内的并发控制通过 JavaScript 同步特性自然保证。若扩展到多进程/多实例，需要引入数据库行锁或分布式锁。

### 3.3 赔付责任方

| 异常类型 | 赔付方向 | 责任方 |
|----------|----------|--------|
| 丢件 | 赔付客户 | 物流/仓库 |
| 破损 | 赔付客户 | 物流/仓库 |
| 短少 | 补货 | 仓库补发 |
| 错件 | 换货 | 仓库换发 |

**留白说明**：赔付金额、赔付比例、保险理赔等高级功能暂未实现，当前仅记录赔付方向和工单金额。

### 3.4 数据一致性保证

**假设**：V3 的工单/审批/扫描/赔付记录之间通过 ticket_id 外键关联，数据完整性由数据库事务保证。

**留白说明**：当前已使用 Supabase PostgreSQL 持久化，但跨系统的数据一致性（V2 与 V3 之间）仍依赖接口调用的最终一致性。

### 3.5 大模型辅助

**假设**：引入 AI 辅助品控推荐和审批建议，但 AI 仅为辅助角色，最终决策权在人工。

详见：[大模型调用说明](./大模型调用说明.md)

## 五、如果想问产品经理的问题清单

1. 品控暂扣的 30 分钟超时，如果仓库是三班倒（24 小时作业），夜间是否也需要同样严格的超时？
2. 不同品类的货物破损判定标准是否一致？（生鲜 vs 电子产品的"破损"定义不同）
3. 二级审批人是否可以转交审批权给其他人？
4. 审批人离职后，其待处理工单是自动转交还是需要管理员手动分配？
5. 赔付金额是否需要财务对账环节？还是只做记录不接支付？
6. V2 的运单金额字段如果后续从 int 改为 decimal，V3 是否需要兼容？
7. 品控主管的快速放行是否需要一个"冷却期"？（比如同一品控主管 1 小时内最多快速放行 N 次）
8. 不同类型异常（如品控工单未关闭 + 物流丢件）是否允许同时存在？

## 六、单租户假设说明

当前 V2 数据不区分租户/商户，因此 V3 采用**单租户假设**：
- 所有上报人对所有运单都有上报权限
- 审批人可以看到所有待审批工单

**如果要支持多租户**：
- `exception_tickets` 表增加 `tenant_id` 字段
- `waybill_snapshots` 表增加 `tenant_id` 字段
- 上报人只能上报自己租户下的运单（查询加 `tenant_id` 过滤）
- 审批人只能看到自己租户的待审批工单
- V2 接口调用时传入 `tenant_id` 进行数据隔离

## 七、考试环境简化说明

在本次考试中，以下方面做了合理简化：

1. 用户认证使用简单的角色字符串，不依赖真实登录系统
2. 工单/审批/扫描数据存储在内存中，重启丢失（符合考试环境的一次性测试需求）
3. 运单快照存储在数据库中（Supabase PostgreSQL），验证跨系统数据同步能力
4. 审批流程采用简化的二级审批（非多级多部门审批）
5. 赔付联动仅记录方向和金额，不涉及实际财务操作
