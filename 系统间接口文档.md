# 系统间接口文档

## V3 ↔ V2 接口契约

---

## 一、接口概述

V3（运单全流程管理系统）通过 HTTP API 与 V2（万能导入系统）交互。两个系统独立部署、独立数据库，通过 RESTful API 完成数据交换。

| 项目 | 说明 |
|------|------|
| V2 接口基地址 | `https://your-v2-project.vercel.app` |
| 鉴权方式 | Bearer Token（HTTP Header: `Authorization: Bearer <API_KEY>`） |
| 请求格式 | JSON |
| 响应格式 | JSON |
| 超时时间 | 10 秒 |
| 重试策略 | 最多重试 2 次，间隔递增（1s、2s） |

---

## 二、接口列表

### 2.1 SKU 校验接口

校验指定 SKU 是否归属于指定运单，用于扫描品控时的真实性校验。

```
GET /api/waybills/verify-sku?external_code={运单号}&sku_code={SKU编码}

Headers:
  Authorization: Bearer <API_KEY>
  X-Request-ID: <唯一请求ID>

Response 200:
{
  "valid": true,
  "waybill_id": "wb_xxx",
  "reason": "SKU 不属于该运单"  // 仅在 valid=false 时返回
}

Response 401:
{ "error": "未授权" }
```

---

### 2.2 运单同步接口

供 V3 拉取运单数据到本地快照表。支持全量拉取和按运单号筛选。

```
POST /api/waybills/sync

Headers:
  Authorization: Bearer <API_KEY>
  X-Request-ID: <唯一请求ID>

Request Body:
{
  "external_codes": ["YD001", "YD002"]  // 可选，不传则返回最近100条
}

Response 200:
[
  {
    "id": "wb_xxx",
    "external_code": "YD001",
    "store_name": "门店A",
    "receiver_name": "张三",
    "receiver_phone": "13800138000",
    "receiver_address": "XX市XX区XX路",
    "amount": 150.00,
    "created_at": "2026-07-04T10:00:00Z",
    "items": [
      {
        "id": "item_xxx",
        "waybill_id": "wb_xxx",
        "sku_code": "SKU001",
        "sku_name": "商品A",
        "quantity": 5,
        "spec": "500ml"
      }
    ]
  }
]
```

### 2.3 异常状态回写接口

V3 通知 V2 某运单存在未关闭异常工单，V2 侧可展示标记。

```
POST /api/waybills/exception-status

Headers:
  Authorization: Bearer <API_KEY>
  X-Request-ID: <唯一请求ID>

Request Body:
{
  "external_code": "YD001",
  "has_open_ticket": true
}

Response 200:
{ "success": true }
```

查询异常状态：
```
GET /api/waybills/exception-status?external_code=YD001

Response 200:
{
  "has_open_ticket": true,
  "updated_at": "2026-07-04T16:00:00Z"
}
```

## 三、鉴权机制

| 机制 | 说明 |
|------|------|
| 方式 | Bearer Token |
| Token | 通过环境变量 `INTERNAL_API_KEY`（V2）和 `V2_API_KEY`（V3）配置 |
| 校验 | V2 每个接口都校验 `Authorization` header |
| 安全性 | 非企业级 OAuth，但满足内部系统间调用的基本安全要求 |

---

## 四、超时与重试策略

| 参数 | 值 | 说明 |
|------|-----|------|
| 请求超时 | 10 秒 | `V2_REQUEST_TIMEOUT` 环境变量 |
| 最大重试次数 | 2 次 | `V2_MAX_RETRIES` 环境变量 |
| 重试间隔 | 递增（1s, 2s） | 避免瞬时故障导致的重试风暴 |
| 幂等性保证 | Request ID 去重 | 每次请求生成唯一 `X-Request-ID`，V3 侧记录日志防止重复处理 |

---

## 五、V2 服务不可用时的降级方案

| 场景 | V3 行为 |
|------|---------|
| V2 接口超时 | 显示"数据获取超时，请稍后重试"，自动重试 2 次 |
| V2 接口返回 5xx | 同超时处理，记录错误日志 |
| V2 整体不可用 | 降级使用本地快照表数据，前端标注"数据可能非最新，获取自 XX 时间" |
| V2 恢复后 | 自动恢复实时调用，本地快照在下次定时同步时更新 |

**降级不降功能**：即使 V2 不可用，V3 的工单查询、审批、执行联动等核心功能仍可基于本地快照数据正常运行。只是新增扫描/上报时的运单校验可能失败，需提示用户稍后再试。

## 六、链路追踪

每次跨系统调用生成唯一 `X-Request-ID`，写入 `sync_logs` 表：

| 日志字段 | 说明 |
|----------|------|
| request_id | 唯一请求 ID，可追踪完整调用链 |
| endpoint | 调用的接口路径 |
| method | HTTP 方法 |
| params_summary | 入参摘要 |
| status_code | 响应状态码 |
| success | 是否成功 |
| duration_ms | 耗时（毫秒） |
| error_message | 错误信息（区分 404/500/超时等） |

---

## 七、老系统二开意识

### V2 原本没有对外接口时的处理方案

如果 V2 原本没有现成的对外 API：

1. **接口版本策略**：新接口使用 `/api/waybills/` 路径前缀，与 V2 原有内部接口隔离
2. **字段向后兼容**：新增字段使用可选类型，不删除现有字段，不改变现有字段类型
3. **灰度上线**：先在测试环境验证，再通过环境变量控制接口是否暴露
4. **接口文档同步**：接口变更时同步更新本文档

### V2 接口字段升级的兼容处理

如果 V2 的 `amount` 字段从 `int` 改为 `decimal`：

1. V3 使用 `parseFloat()` 读取金额字段，天然兼容 int/decimal 变化
2. 快照表使用 `NUMERIC` 类型存储，兼容两种精度
3. 如果字段名发生变化，V3 通过配置映射（类似 V2 的字段映射机制）进行适配
4. V3 的接口同步日志会记录响应结构，便于排查字段不匹配问题
