# 反思题

> V3 运单异常工单审批系统 — 考试反思题

---

## 1. **状态机设计的优缺点**

**问题**：当前工单状态机采用 `pending → level2 → approved → closed` 的四状态设计，请分析这种设计的优缺点。

**回答要点**：
- **优点**：状态数量少，流转清晰；金额阈值驱动二级审批，规则简单；拒绝后回到 pending 支持重提。
- **缺点**：
  - 状态抽象层次不够。比如 `pending` 和 `level2` 实际都是"审批中"的子状态，但平铺在一起。
  - 缺少"撤回"状态。上报人提交后无法主动撤回工单。
  - 缺少"挂起/暂缓"状态。实际业务中有些工单需要等待证据补充。
  - 状态机未持久化到数据库，服务重启后状态丢失。
- **改进建议**：引入"活动状态"和"终态"的概念分层；添加 `suspended`（挂起）、`withdrawn`（撤回）、`archived`（归档）状态。

## 2. **并发控制的充分性**

**问题**：当前系统通过内存状态机的同步性来实现并发控制。你认为这种方案在什么场景下会失效？如何改进？

**回答要点**：
- **失效场景**：
  - 多进程/多实例部署时，内存状态不共享，两个不同进程可以同时审批同一个工单。
  - Node.js 虽然是单线程事件循环，但 `await` 会让出控制权，如果两个请求在同一个 tick 内读取到相同状态，仍可能产生竞态条件。
- **改进方案**：
  - 短期：引入 Node.js `async-mutex` 锁，按 ticket_id 做互斥。
  - 长期：工单持久化到 PostgreSQL，使用 `SELECT ... FOR UPDATE` 行锁，配合乐观锁版本号（`version` 字段）。
  - 分布式：引入 Redis 分布式锁（Redlock）或使用数据库事务隔离级别 SERIALIZABLE。

## 3. **赔偿方向的合理性**

**问题**：当前系统在 `tickets/store.ts` 和 `v3-store.ts` 中对同一异常类型的赔付方向定义不一致（如 `damaged` 在 v3-store 中为"核销"而在 tickets/store 中为"赔付客户"）。你认为哪个更合理？为什么？

**回答要点**：
- **赔付客户更合理**（`tickets/store.ts` 的选择）：破损通常是物流过程中的责任，应由物流方赔付客户，再内部追责。从客户体验角度出发，破损件应当"先赔付客户"。
- **核销也有道理**（`v3-store.ts` 的选择）：如果破损商品经过品控确认可以打折销售或内部消化，走核销流程更符合成本控制逻辑。
- **正确做法**：应当根据破损责任归属（物流责任 / 商家责任 / 客户责任）动态确定赔付方向，而非一刀切。责任归属字段可以作为一个扩展点。

## 4. **内存状态机 vs 数据库持久化**

**问题**：当前 V3 将工单、审批记录、扫描记录存储在内存中。如果系统部署到生产环境，你认为必须将哪些数据持久化到数据库？为什么？

**回答要点**：
- **必须持久化的数据**：
  - **工单** — 核心业务实体，需要审计追踪
  - **审批记录** — 审批链需要不可篡改的留痕
  - **扫描记录** — 品控过程需要溯源
  - **赔付记录** — 涉及财务，必须有持久化保障
- **可以保留在内存中的数据**：
  - 计数器（如 `retry_count`）— 可以每次从数据库计算
  - 临时锁状态 — 可以用 Redis 处理
- **迁移策略**：先建表 → 双写（内存+数据库）→ 灰度切读 → 下线内存存储。

## 5. **快照降级策略的边界条件**

**问题**：V3 的运单数据获取采用"V2 实时拉取 → 本地快照 → 硬编码 fallback"三级降级。请分析各层降级的边界条件和失效风险。

**回答要点**：
- **第一层 V2 实时拉取**：
  - 失效条件：V2 服务宕机、网络分区、超时（8秒）、鉴权失败
  - 风险：V2 部分可用时（连接成功但查询超时）无法降级（当前实现中只有在 `catch` 块才降级）

- **第二层 本地快照**：
  - 失效条件：快照表不存在、首次同步未完成就 V2 故障、快照数据过期
  - 风险：**数据时效性问题** — 快照是上次同步时的数据，V2 新增的运单无法反映。建议增加 `synced_at` 过期策略。

- **第三层 硬编码 fallback**：
  - 失效条件：代码中的测试数据不匹配实际业务
  - 风险：**数据准确性为零** — 硬编码运单与实际运单完全无关，可能导致工单创建失败或数据混乱。

- **改进建议**：
  - 快照增加 TTL（如 24 小时过期）
  - V2 连接成功但返回错误的中间态也要降级（不只是 catch）
  - 硬编码 fallback 应该至少返回空数组而非假数据，让上层感知到数据不可用

## 6. **权限模型的安全隐患**

**问题**：当前系统通过 `operator.includes("supervisor")` 判断放行权限。请分析这种权限模型的安全隐患。

**回答要点**：
- **安全隐患**：
  - **字符串包含匹配过于宽松**：任何包含 "supervisor" 子串的名字都能通过，如 `"not_a_supervisor_but_hacker"` 也能放行。
  - **无签名验证**：前端传入的 `operator` 字符串可以被任意伪造。
  - **无角色注册机制**：没有用户-角色的映射表，无法管理权限变更。
- **改进方案**：
  - 使用基于角色的访问控制（RBAC），用户-角色-权限三层模型
  - JWT Token 中携带用户角色，服务端解析 Token 获取角色
  - 配置 `allowedRoles: string[]` 用精确匹配替代字符串包含

## 7. **测试覆盖的完整性**

**问题**：`exam-test-v3.mjs` 覆盖了 9 个考点、50+ 测试项。你认为还有哪些场景未被覆盖？

**回答要点**：
- **未覆盖场景**：
  - 大并发下的数据一致性（100+ 并发审批同一工单）
  - 审批超时机制（需先实现超时功能）
  - 重提次数达上限后的行为（当前无上限）
  - V2 部分可用时的边界行为（连接建立但 SQL 超时）
  - 快照数据与 V2 数据不一致时的冲突处理
  - 多运单同时异常时的批量工单处理
  - 赔付金额计算与工单金额不一致的场景
- **建议补充**：添加混沌工程测试（随机注入网络故障、延迟），验证降级策略的鲁棒性。

## 8. **系统可观测性**

**问题**：V3 目前通过 `/api/monitor` 暴露基本的计数指标。你认为还需要哪些可观测性手段？

**回答要点**：
- **必要补充**：
  - **结构化日志**：每次 API 调用携带 `request_id`（追踪ID），贯穿 V2→V3 全链路。
  - **指标收集**：QPS、P50/P95/P99 延迟、错误率、审批流转时长分布。
  - **告警规则**：快照过期超过 N 小时告警、V2 不可用超过 M 分钟告警、审批积压告警。
  - **链路追踪**：OpenTelemetry + Jaeger/Zipkin，追踪单个工单从创建到审批完成的全链路时间。
- **考试环境简化**：本次考试中已实现 `request_id` 追踪和基本计数指标，满足考试要求。
