# 需求理解与假设说明

> V3 运单异常工单审批系统 — 需求理解、假设与留白点说明

---

## 一、需求理解

### 1.1 系统定位

V3 系统是一个面向物流/仓储场景的 **运单异常工单管理与审批系统**，作为 V2（万能导入系统）的下游消费端，负责将 V2 导入的运单数据进行品控扫描、异常识别、多级审批和赔付联动处理。

### 1.2 核心业务场景

| 场景 | 说明 |
|------|------|
| **手动上报异常** | 仓管/品控人员发现运单异常（丢件、破损、短少、错件）后手动创建工单 |
| **扫描品控** | 扫描枪扫码核对 SKU 数量/规格/破损情况，不通过时自动创建工单并暂扣 |
| **分级审批** | 低金额工单一级审批直达，高金额（>500元）需二级审批 |
| **异常回写** | 审批结果回写 V2 系统，标记运单存在未闭环异常工单 |
| **赔付联动** | 审批通过后自动生成赔付/补货/换货方向记录 |

### 1.3 与 V2 的关系

V2 是"万能导入系统"，负责文件智能解析、运单数据入库。V3 是"异常管理审批系统"，通过 API 接口从 V2 拉取运单数据到本地快照，独立完成后续的工单管理流程。

---

## 二、关键假设与留白点

### 2.1 分级审批阈值

**假设**：单笔工单金额 **> 500 元** 自动进入二级审批（`level2`），需要两名不同级别的审批人依次通过。

**理由**：参考物流行业常见的分权审批实践，500 元是一个合理的分界线，平衡了效率与风控。

**可配置方向**：阈值可设计为环境变量 `APPROVAL_LEVEL2_THRESHOLD`，方便不同客户调整。

### 2.2 审批超时时长

**假设**：审批操作无超时限制，工单可在任意时间被审批。

**留白说明**：实际业务中，超时未审批的工单可能需要自动升级或提醒。当前版本未实现超时机制，后续可按需添加：
- 一级审批超时自动升级到二级审批
- 超时工单通知管理员

### 2.3 重提次数

**假设**：工单被拒绝后，`retry_count` 递增，但 **不限制重提次数上限**。

**留白说明**：生产环境中通常有最多重提 3 次的限制，超过后需管理员介入处理。当前版本保留了 `retry_count` 计数器，但未添加硬上限。

### 2.4 物流异常类型

**假设**：覆盖 4 种核心异常类型，足以满足大多数物流场景。

| 异常类型 | 含义 | 赔付方向 |
|----------|------|----------|
| `lost` | 丢件 | 赔付客户 |
| `damaged` | 破损 | 赔付客户 |
| `shortage` | 短少 | 补货 |
| `wrong_item` | 错件 | 换货 |

**留白说明**：其他异常类型（如延期、包装损坏、客户拒收等）暂未覆盖，可通过扩展异常类型枚举添加。

### 2.5 角色权限

**假设**：基于操作人标识字符串进行权限校验，无需完整的登录/认证系统。

| 角色 | 权限 |
|------|------|
| `admin` | 所有操作 |
| `approver_level1_*` | 一级审批 |
| `approver_level2_*` | 二级审批 |
| `qc_supervisor_*` | 品控主管放行 |
| `operator_*` | 扫描操作 |
| `test_reporter_*` | 上报异常 |

**留白说明**：当前版本使用简单的字符串匹配实现权限校验（如 `operator.includes("supervisor")`），未接入真实的用户认证系统。在考试环境中这是合理的简化假设，生产环境需接入 JWT/OAuth 等标准认证。

### 2.6 数据同步与快照

**假设**：V3 通过主动拉取（Pull）方式从 V2 获取运单数据，首次同步成功后自动缓存到本地 `waybill_snapshots` 表。

**降级策略**：
1. **优先 V2 实时拉取**（8秒超时）
2. **快照兜底**（V2 不可用时读取本地快照）
3. **硬编码 fallback**（快照也无数据时的最后保障）

**留白说明**：当前为单向同步（V2→V3），V3 的审批结果通过 `/api/waybills/exception-status` 回写到 V2。真实场景可能需要双向同步或消息队列保证最终一致性。

### 2.7 品控规则

**假设**：扫描判定基于三个维度：数量匹配、规格匹配、破损等级。

```
match = (actual_qty === expected_qty) && spec_match && (damage_level === 0)
```

- `match === true` → 扫描通过 (`pass`)
- `match === false` → 品控暂扣，自动创建工单 (`fail`)

**留白说明**：品控规则当前是硬编码的，未支持动态配置（如按客户/门店设置不同的容差率）。可扩展为规则引擎配置表。

### 2.8 并发控制

**假设**：审批操作需要互斥保护，同一工单不能同时被两人审批。

**实现**：使用幂等性检查（同一审批人重复操作返回 409）+ 状态机互斥（已完结工单不可再操作）。

**留白说明**：当前使用内存状态机，单进程内的并发控制通过 JavaScript 同步特性自然保证。若扩展到多进程/多实例，需要引入数据库行锁或分布式锁。

### 2.9 赔付责任方

| 异常类型 | 赔付方向 | 责任方 |
|----------|----------|--------|
| 丢件 | 赔付客户 | 物流/仓库 |
| 破损 | 赔付客户 | 物流/仓库 |
| 短少 | 补货 | 仓库补发 |
| 错件 | 换货 | 仓库换发 |

**留白说明**：赔付金额、赔付比例、保险理赔等高级功能暂未实现，当前仅记录赔付方向和工单金额。

### 2.10 数据一致性保证

**假设**：V3 的工单/审批/扫描/赔付记录之间通过 ticket_id 外键关联，数据完整性由内存状态机保证。

**留白说明**：当前工单、审批记录、扫描记录存储在内存中，服务重启后数据丢失。生产环境需持久化到数据库，并添加事务保证。

## 三、未覆盖项（已知局限）

| 项目 | 状态 | 说明 |
|------|------|------|
| 用户认证系统 | 未实现 | 使用字符串匹配模拟 |
| 审批超时 | 未实现 | 工单可无限等待 |
| 重提次数上限 | 未实现 | retry_count 计数但无上限 |
| 赔付金额计算 | 简化 | 直接使用工单金额 |
| 通知推送 | 未实现 | 无邮件/短信/站内通知 |
| 数据持久化 | 部分 | 工单在内存中，快照在数据库 |
| 审计日志 | 未实现 | 仅有审批记录 |
| 批量操作 | 部分 | 支持批量创建，不支持批量审批 |
| 报表统计 | 未实现 | monitor 接口仅返回基本计数 |
| 移动端适配 | 未实现 | 未做响应式设计 |

## 四、考试环境简化说明

在本次考试中，以下方面做了合理简化：

1. 用户认证使用简单的角色字符串，不依赖真实登录系统
2. 工单/审批/扫描数据存储在内存中，重启丢失（符合考试环境的一次性测试需求）
3. 运单快照存储在数据库中（Supabase PostgreSQL），验证跨系统数据同步能力
4. 审批流程采用简化的二级审批（非多级多部门审批）
5. 赔付联动仅记录方向和金额，不涉及实际财务操作
